🪜 1. Interacciones entre Servicios

Interacción 1

Servicio de Usuarios
Servicio de Autenticación
Credenciales de acceso (usuario y contraseña)
El usuario no podrá acceder al sistema. Se debe mostrar un mensaje de error y permitir reintentos.

Interacción 2

Servicio de Páginas Web
Servicio de Inteligencia Artificial
Solicitud de generación automática de descripciones o contenido
No se generarán descripciones automáticas. El usuario deberá crear el contenido manualmente. El sistema debe continuar funcionando sin la IA.

Interacción 3

Servicio de Productos
Servicio de Imágenes (Cloudinary)
Archivos de imágenes para productos
No se podrán subir ni visualizar imágenes. El sistema debe mostrar un mensaje de error controlado y permitir continuar sin imágenes temporalmente.

🪜 2. REST o Eventos

Interacción REST Evento ¿Por qué?
Usuarios→ autenticacion ✓ Requiere respuesta inmediata para validar credenciales y permitir acceso al sistema
Paginas web→ Bases de Datos ✓ Es una solicitud directa de contenido que requiere una respuesta específica
Paginas web→IA/Imagenes ✓ Es una operación de carga de archivos que requiere confirmación inmediata del almacenamiento

🪜 3. Síncrono o Asíncrono

Interacción Síncrono Asíncrono Justificación
autenticacion ✓ El sistema debe esperar la respuesta de autenticación antes de permitir el acceso. Es crítico para la seguridad.
Bases de Datos ✓ La generación de contenido con IA puede tomar tiempo. El sistema puede continuar mientras procesa la respuesta en segundo plano.
IA/Imagenes ✓ Se necesita confirmar que la imagen se subió correctamente antes de continuar con la creación del producto.

🪜 4. Eventos Importantes del Sistema

1️⃣ Usuario registrado: Cuando un nuevo usuario crea su cuenta, se dispara un evento que permite inicializar su perfil y configurar su espacio de trabajo.

2️⃣ Página web publicada: Cuando un usuario publica su página web, se genera un evento que activa el despliegue en Vercel y notifica al sistema de páginas web.

3️⃣ Producto creado: Cuando se crea un nuevo producto, se dispara un evento que puede activar la generación automática de descripciones mediante el servicio de IA.

🪜 5. Análisis de Fallos

Servicio de Orquestación de Páginas Web Nomo
En un sistema distribuido, si falla el orquestador principal, se pierde el cumplimiento del propósito de la aplicación. Aunque el usuario pueda ver su perfil o autenticarse, no podrá cumplir la función vital de "Generar o Publicar Sitios", dejando al sistema sin su valor principal.
El sistema debe continuar de forma degradada. Se deben permitir micro-tareas como edición de contenido listo para ser guardado localmente o visualización de estadísticas, pero la publicación y generación quedarán deshabilitadas hasta que el orquestador se recupere.
Aplicar el patrón Circuit Breaker para evitar que otros servicios como el de IA sigan enviando peticiones inútiles, usar colas de eventos (RabbitMQ) para encolar solicitudes de publicación y procesarlas asincrónicamente cuando el servicio crítico retorne, y tener redundancia de nodos orquestadores.

🪜 6. Service Discovery

Los servicios que dependen de ese servicio dejarían de poder comunicarse con él, causando fallos en cascada. Tendríamos que actualizar manualmente todas las configuraciones en cada servicio que lo consume.
Actualmente, los servicios están configurados con URLs fijas (Supabase, Cloudinary, Groq). Con Service Discovery, los servicios se registrarían automáticamente y otros servicios podrían encontrarlos dinámicamente mediante un registro centralizado.
Permitiría escalar servicios independientemente, manejar fallos automáticamente redirigiendo a instancias saludables, y facilitar el despliegue de nuevas versiones sin interrumpir el servicio. Especialmente útil cuando se tienen múltiples instancias del mismo servicio.
Sería útil para la comunicación entre el Servicio de Usuarios y el Servicio de Autenticación, entre el Servicio de Páginas Web y el Servicio de IA, y para gestionar múltiples instancias del Servicio de Productos cuando se necesite escalar.
