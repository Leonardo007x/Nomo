🪜 2. REST o Eventos

Interacción REST Evento ¿Por qué?
1 (Usuarios → Autenticación) ✓ Es una petición directa: el usuario envía sus credenciales y necesita una respuesta inmediata para saber si puede entrar o no al sistema.
2 (Páginas/Productos → Base de Datos) ✓ Consultas y operaciones directas a la base de datos. Se hace una petición y se espera obtener o confirmar los datos almacenados.
3 (Imágenes → Cloudinary) ✓ Subida de archivos mediante petición HTTP. El servicio envía el archivo y espera la confirmación y la URL de la imagen subida.

🪜 3. Síncrono o Asíncrono

Interacción Síncrono Asíncrono Justificación
1 (Autenticación) ✓ El usuario debe esperar la verificación para continuar. No tendría sentido dejarlo avanzar sin saber si sus credenciales son válidas.
2 (Base de Datos) ✓ Se requiere la información para mostrar la página o confirmar que se guardó correctamente. La operación debe completarse antes de seguir.
3 (Cloudinary) ✓ Aunque teóricamente podría ser asíncrona, en nuestra implementación el usuario espera la confirmación de la subida para poder usar la imagen en su sitio.

🪜 4. Eventos Importantes del Sistema

1️⃣ Usuario inicia sesión: El sistema verifica las credenciales y permite o deniega el acceso al panel del negocio.

2️⃣ Se crea o se publica una página web: La información se guarda en la base de datos y, si corresponde, se despliega en Vercel para que los visitantes puedan verla.

3️⃣ Se genera contenido con IA: Cuando el usuario solicita una descripción automática para un producto, el servicio de IA (Groq) procesa la petición y devuelve el texto generado.

🪜 5. Análisis de Fallos

Lo más crítico no es un componente técnico aislado, sino el fin de la aplicación: que los negocios puedan ofertar sus productos a los clientes. En un sistema distribuido, el servicio más crítico es aquel que cumple ese propósito: el flujo que permite publicar y mostrar las páginas web con el catálogo de productos a los visitantes (combinación de Páginas Web + Productos + el despliegue público). Si eso falla, Nomo pierde su razón de ser.
Si falla el flujo que permite ofertar productos a los clientes (por ejemplo, por caída de la base de datos, del servidor o de los servicios de páginas y productos), los visitantes no podrían ver las tiendas ni los artículos. Los negocios perderían su presencia digital y las oportunidades de venta. Aunque otros servicios sigan funcionando, el propósito central de la plataforma quedaría afectado.
Cuando falla un servicio crítico, lo ideal es que la aplicación no se caiga por completo sino que muestre mensajes de error comprensibles al usuario, intente reintentar la operación y sugiera intentar más tarde. Si es posible, algunas funciones secundarias (como visualizar páginas ya cargadas en caché) podrían seguir funcionando.
Implementar reintentos automáticos, mostrar mensajes de error claros, realizar copias de seguridad periódicas de la base de datos y monitorear constantemente el estado del sistema para detectar fallos cuanto antes.

🪜 6. Service Discovery

Los demás servicios no sabrían cómo contactarlo. Si un microservicio cambia de dirección, las peticiones fallarían porque seguirían apuntando a la dirección antigua, lo que provocaría errores y posiblemente caídas en la comunicación entre componentes.
Mediante un registro centralizado de servicios: cada servicio se registra al iniciar y actualiza su información. Cuando otro necesita comunicarse con él, consulta ese registro para obtener la dirección actual (IP, puerto o URL) sin tenerla hardcodeada.
Permite que los servicios se encuentren de forma dinámica cuando hay múltiples instancias, despliegues en distintos entornos o cambios de infraestructura. Evita tener que actualizar manualmente las direcciones en cada servicio y facilita la escalabilidad y el mantenimiento.
Sería especialmente útil si en el futuro Nomo tuviera más microservicios desplegados de forma dinámica (por ejemplo, varias instancias del servicio de autenticación o del de IA para distribuir carga). En la arquitectura actual con Vercel, Supabase y Cloudinary, las URLs son relativamente fijas, pero Service Discovery ayudaría si se añadieran más servicios propios que puedan cambiar de ubicación o replicarse.
