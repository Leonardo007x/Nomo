🧩 Contexto del proyecto (Nomo)

Nomo ayuda a pequeñas y medianas empresas que no tienen presencia digital a crear y personalizar un sitio web con plantillas prediseñadas, sin necesitar conocimientos de programación.
Emprendedores, pequeñas empresas, comercios locales y personas que venden bienes o servicios.
Muchas empresas seguirían sin presencia en línea, tendrían que contratar un desarrollador, perderían oportunidades de comercio electrónico y su crecimiento sería más lento frente a negocios digitalizados.

🧱 Servicios identificados (microservicios)

Usuarios: creación de cuentas, perfiles.
Autenticación: verificación de credenciales, permisos.
Páginas Web: creación/edición del sitio, plantillas, publicación.
Productos: CRUD de artículos y vínculo con el sitio.
Imágenes: carga/gestión en Cloudinary.
Inteligencia Artificial: generación de descripciones y apoyo de contenido (Groq).

🪜 1. Interacciones entre Servicios

Interacción 1

Servicio de Usuarios / Frontend
Servicio de Autenticación
Credenciales (usuario y contraseña)
El usuario no podría iniciar sesión ni acceder al sistema.

Interacción 2

Servicio de Páginas Web
Base de datos (Supabase)
Configuraciones de la página, detalles del negocio y datos a almacenar
No se podría almacenar ni consultar información; las páginas no podrían guardarse ni desplegarse.

Interacción 3

Servicio de Imágenes / Páginas Web
Cloudinary (almacenamiento de imágenes)
Archivos multimedia (imágenes) para productos y páginas
No se podrían subir ni visualizar imágenes; el usuario tendría que usar solo texto.

Interacción 4

Servicio de Páginas Web / Productos
Servicio de Inteligencia Artificial (Groq)
Texto base o atributos del producto/negocio para generar descripciones o contenido
No se generarían descripciones automáticas; el usuario tendría que escribirlas manualmente (degradación del sistema, pero puede continuar).

Interacción 5

Servicio de Productos
Base de datos (Supabase)
Productos (crear/editar/eliminar), relación con el sitio y metadatos
No se podrían guardar cambios de productos; el catálogo no se actualizaría y podrían perderse operaciones.

🪜 2. REST o Eventos

Interacción REST Evento ¿Por qué?
Usuarios → Autenticación ☑ ☐ Es una petición directa: el cliente envía credenciales y espera una respuesta (token/sesión).
Páginas Web → Base de Datos ☑ ☐ Consultas y escrituras a la BD son operaciones request-response (Supabase API REST).
Páginas Web/Productos → IA (Groq) ☑ ☐ Se envía un prompt/atributos y se espera una respuesta con texto generado (descripciones, apoyo de contenido).
Páginas Web/Productos → Imágenes (Cloudinary) ☑ ☐ Se sube un archivo y se recibe una URL; es request-response aunque la carga pueda ejecutarse sin bloquear la UI.

🪜 3. Síncrono o Asíncrono

Interacción Síncrono Asíncrono Justificación
Autenticación ☑ ☐ El usuario debe esperar la verificación para poder continuar; sin respuesta no hay sesión.
Base de Datos ☑ ☐ Guardar y leer datos es necesario de forma inmediata para mostrar o actualizar la página.
Subida de Imágenes (Cloudinary) ☐ ☑ La carga puede tardar; conviene no bloquear la interfaz y manejar progreso/reintentos.
Generación de contenido (IA / Groq) ☑ ☐ Normalmente el usuario espera el texto para completar el formulario; si falla, se puede degradar a edición manual.

🪜 4. Eventos Importantes del Sistema

1️⃣ Usuario se registra / inicia sesión — Se crean credenciales, se verifican y se habilita sesión/permisos.

2️⃣ Usuario crea o edita una página web — Se guarda configuración y detalles del negocio en Supabase; si hay imágenes, se suben a Cloudinary; si requiere contenido, consulta a Groq.

3️⃣ Usuario gestiona productos — Se agregan/modifican/eliminan artículos y se sincronizan con la BD.

4️⃣ Usuario publica la página — La página se despliega en Vercel y queda disponible para visitantes.

🪜 5. Análisis de Fallos

Servicio de Páginas Web. Es el servicio que entrega el valor principal: permitir crear/editar/publicar el sitio del negocio y coordinar plantillas, productos, imágenes e IA.
Los usuarios no podrían construir ni actualizar su página (ni aplicar plantillas, ni publicar cambios). La plataforma perdería su funcionalidad principal aunque otros servicios estén disponibles. Dependiendo del tipo de fallo, podría quedar solo lectura (ver páginas ya publicadas) pero sin edición/gestión desde Nomo.
Debe continuar con degradación controlada: habilitar navegación/visualización cuando sea posible y deshabilitar únicamente acciones de edición/publicación, mostrando un estado claro. Además, si fallan dependencias (Groq/Cloudinary/BD), el servicio core debe seguir operando con funciones reducidas (descripción manual, placeholders, borradores).
Balanceo y réplicas del servicio de Páginas Web; health checks y auto-restart; circuit breaker/timeouts hacia dependencias; colas para tareas pesadas (subida de imágenes/IA); caché de plantillas/configuraciones y modo borrador local; monitoreo y alertas. (La BD se protege con backups, pero como dependencia, no como el “core” del sistema).

🪜 6. Service Discovery

Los demás servicios que dependen de él no podrían conectarse; las solicitudes fallarían y habría errores de conexión. Habría que actualizar manualmente las URLs en cada cliente.
Por configuración estática (URLs hardcodeadas), variables de entorno, o mediante Service Discovery que mantiene un registro dinámico de las ubicaciones y estados de los servicios.
Permite que los servicios se encuentren automáticamente cuando cambian IPs, puertos o se añaden réplicas; facilita la escalabilidad independiente y el despliegue en entornos como contenedores o Kubernetes.
Principalmente entre microservicios internos si la arquitectura crece (por ejemplo, separar IA, imágenes o autenticación como servicios propios con múltiples instancias). Con servicios externos (Supabase, Groq, Cloudinary) normalmente se usan URLs estables + variables de entorno, pero un registro/descubrimiento ayuda si se migran a instancias propias o se agrega balanceo dinámico.
